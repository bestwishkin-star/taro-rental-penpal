# 真实租房经历社区设计

**日期**: 2026-05-09
**状态**: 已确认方向，待拆分实现计划

## 背景

当前“我要分享房源”的产品表达偏向房屋中介：用户填写房源价格、区域、户型、面积、联系方式和图片，详情页也更像房源卡片。这会让产品目标从“真实租房经验交流”偏移到“找房信息撮合”。

新的目标是把分享功能重构为租房垂直领域的社区内容：用户分享自己的真实租房经历、避坑经验、区域感受、租金参考和生活体验。产品形态参考小红书笔记，但细分领域限定为“租房经历”。

## 产品定位

第一版定位为“强调真实性的租房经历社区”，不是房源中介发布工具。

核心原则：

- 内容主体是“租房经历笔记”，不是“待出租房源”。
- 房源信息只作为经历的事实补充，用于帮助其他用户判断参考价值。
- 用户可以公开评论、追问和补充信息，形成社区讨论。
- 真实性通过发布承诺、可选凭证、专项举报和后台复核闭环建立。
- 不强调“立即联系看房”“复制微信”等中介导向动作。

## 第一版范围

### 包含

- “我要分享房源”改为“分享租房经历”。
- 发布页以笔记内容为主，结构化租房信息为辅。
- 发布时支持可选真实性凭证，凭证不公开展示。
- 列表页从房源列表改为租房经历笔记流。
- 详情页突出经历正文、图片、标签、租房参考信息和评论区。
- 支持公开评论。
- 支持租房专项举报。
- 后台支持查看举报、查看凭证和处理内容状态。

### 暂不包含

- 私信能力。
- 精确地图选点。
- 地理位置授权。
- 平台官方认证真伪。
- 自动审核模型。
- 复杂用户信用分。
- 支付、担保、合同或看房预约。

## 用户角色

### 分享者

分享自己的租房经历，包括真实居住感受、避坑点、区域体验、租金参考和图片。可以选择提交凭证帮助平台处理真实性争议。

### 浏览者

通过笔记流浏览租房经验，按城市、区域、租金和标签筛选内容，在详情页评论追问或补充信息。

### 举报者

发现虚假租金、虚假地址、中介伪装、钓鱼引流、隐私泄露或恶意攻击时，可以提交专项举报。

### 管理员

处理举报，查看笔记内容、举报原因和发布者提交的凭证，调整内容状态。

- 消息或评论：后续可承接评论通知。
- 我的：我的分享、收藏、举报记录或处理进度。

## 发布页设计

### 必填字段

| 字段 | 说明 |
| --- | --- |
| 标题 | 用一句话概括经历，例如“在望京合租半年后的真实感受” |
| 正文 | 真实租房经历、优点、缺点、避坑、建议 |
| 图片 | 至少 1 张，建议展示公共区域、周边、房间局部，避免隐私 |
| 城市 / 区域 | 例如北京 / 朝阳 |
| 小区或地标 | 不要求门牌号，例如“望京 SOHO 附近” |
| 租金 | 具体金额或月租参考 |
| 租住阶段 | 例如正在住、已搬离、看房经历、转租经历 |
| 真实性承诺 | 勾选“我承诺内容来自真实租房经历” |

### 可选字段

| 字段 | 说明 |
| --- | --- |
| 户型 | 一居、两居、三居、单间等 |
| 面积 | 用于参考，不作为强制项 |
| 租住类型 | 整租、合租、转租、短租 |
| 通勤信息 | 到地铁、公司、学校或商圈的通勤感受 |
| 标签 | 避坑、推荐、通勤友好、隔音差、采光好、押金争议等 |
| 真实性凭证 | 合同局部、账单、聊天记录、付款记录等，可打码上传 |

### 隐私提示

发布页需要明确提示：

- 不要上传身份证、手机号、银行卡、合同编号、房号、门牌号等敏感信息。
- 上传凭证前请自行打码。
- 凭证仅供平台复核，不会在前台公开展示。

## 列表页设计

列表页改为笔记流，不再呈现为中介房源列表。

卡片信息：

- 封面图。
- 标题。
- 城市 / 区域。
- 租金参考。
- 小区或地标。
- 标签。
- 评论数。
- 收藏数。
- “已提交凭证”标识。

筛选能力：

- 城市。
- 区域。
- 租金范围。
- 租住类型。
- 标签，例如避坑、推荐、通勤、押金、室友、噪音。

排序能力：

- 最新。
- 热门讨论。
- 收藏较多。

## 详情页设计

详情页首屏要弱化“价格卡片”，突出社区内容。

建议结构：

1. 图片轮播。
2. 标题。
3. 作者信息和发布时间。
4. 已提交凭证标识。
5. 经历正文。
6. 标签。
7. 租房参考模块。
8. 评论区。
9. 收藏、评论、举报操作。

租房参考模块展示：

- 城市 / 区域。
- 小区或地标。
- 租金。
- 户型。
- 面积。
- 租住类型。
- 租住阶段。
- 通勤信息。

不再把“联系方式”作为主操作。第一版也不做私信。

## 评论区设计

第一版支持公开评论。

能力范围：

- 发表评论。
- 查看评论列表。
- 删除自己的评论。
- 作者或管理员可以处理不当评论。
- 评论数在列表卡片和详情页展示。

暂不做：

- 楼中楼回复。
- 私信。
- 评论点赞。
- 评论通知聚合页。

## 真实性机制

### 发布承诺

发布时必须勾选真实性承诺：

“我承诺本内容来自我的真实租房经历，并已避免公开他人隐私信息。”

未勾选时不能发布。

### 可选凭证

凭证上传是可选项，但在内容上展示“已提交凭证”标识。

标识文案建议：

- “已提交凭证”
- 辅助说明：“凭证仅供平台复核，不代表官方认证”

凭证使用规则：

- 前台不展示凭证图片。
- 普通用户无法查看凭证。
- 管理员在举报处理或内容复核时可查看。
- 凭证可以为空，未上传不影响发布。

### 内容状态

笔记建议支持以下状态：

| 状态 | 说明 |
| --- | --- |
| 正常 | 前台可见 |
| 待复核 | 被举报或系统标记，前台可继续展示或按策略降权 |
| 已隐藏 | 前台不可见，作者可看到处理提示 |
| 已驳回 | 发布或复核未通过 |
| 作者删除 | 作者主动删除 |

第一版可以先使用最小状态集合：正常、待复核、已隐藏、已驳回。

## 举报设计

举报入口在详情页更多操作中展示，不公开举报数量。

举报原因使用租房专项分类：

- 虚假租金 / 地址。
- 冒充真实经历。
- 中介伪装。
- 钓鱼引流。
- 隐私泄露。
- 恶意攻击。
- 其他。

举报提交字段：

- 举报原因。
- 补充说明。
- 可选图片证据。

举报提交后：

- 用户看到提交成功提示。
- 内容进入后台待处理队列。
- 管理员可以查看笔记、举报原因、举报说明、举报证据和发布者凭证。

## 后台治理

后台需要新增内容治理能力。

### 举报处理列表

展示字段：

- 被举报笔记标题。
- 作者。
- 举报原因。
- 举报次数。
- 最新举报时间。
- 当前内容状态。

### 举报详情

展示内容：

- 笔记正文和结构化租房信息。
- 发布者提交的真实性凭证。
- 举报者提交的说明和证据。
- 管理员处理记录。

### 处理动作

- 标记正常。
- 标记待复核。
- 隐藏内容。
- 驳回内容。
- 记录处理备注。

## 接口建议

### 前台接口

- `GET /api/rental-posts`：笔记流列表。
- `GET /api/rental-posts/:id`：笔记详情。
- `POST /api/rental-posts`：发布租房经历。
- `GET /api/rental-posts/mine`：我的分享。
- `POST /api/rental-posts/:id/favorite`：收藏。
- `DELETE /api/rental-posts/:id/favorite`：取消收藏。
- `GET /api/rental-posts/:id/comments`：评论列表。
- `POST /api/rental-posts/:id/comments`：发表评论。
- `DELETE /api/rental-post-comments/:id`：删除自己的评论。
- `POST /api/rental-posts/:id/reports`：举报笔记。

### 后台接口

- `GET /api/admin/rental-post-reports`：举报列表。
- `GET /api/admin/rental-post-reports/:id`：举报详情。
- `POST /api/admin/rental-post-reports/:id/handle`：处理举报。
- `PATCH /api/admin/rental-posts/:id/status`：调整笔记状态。

## 迁移策略

为了降低改造风险，建议分两步迁移：

1. 兼容当前 rentals 数据表和接口，先在前端表达上从“房源”改为“租房经历”，把 `experience` 提升为主体内容。
2. 再引入新的 post/comment/report/proof 数据结构，把历史 rentals 数据迁移为 rental_posts。

历史数据迁移规则：

- `experience` 映射为 `content`。
- `location` 拆分或保留为 `landmark`。
- `price` 映射为 `rent_amount`。
- `roomType` 映射为 `room_type`。
- `photos` 保持不变。
- `status` 映射到新内容状态。

## 分期建议

### 第一期：社区化发布和展示

- 页面文案改为“分享租房经历”。
- 发布页调整字段结构。
- 列表页改为笔记流。
- 详情页突出经历正文和租房参考。
- 移除联系导向主操作。

### 第二期：评论和举报闭环

- 新增评论表和评论接口。
- 新增举报表和举报接口。
- 详情页接入评论区和举报入口。

### 第三期：凭证和后台复核

- 新增凭证上传和后台查看能力。
- 后台新增举报处理列表和处理动作。
- 前台展示“已提交凭证”标识。

## 验收标准

- 用户看到的发布入口是“分享租房经历”，不会误解为中介发布房源。
- 发布内容必须包含真实经历正文、城市区域、小区或地标、租金和图片。
- 用户可以在笔记详情页评论讨论。
- 用户可以按租房专项原因举报内容。
- 提交凭证后，前台只展示凭证标识，不公开凭证图片。
- 管理员可以基于举报和凭证处理内容状态。
- 页面不依赖地图选点或地理位置权限。
